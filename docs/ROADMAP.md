## R5.5 — Earned-Trust Progression  ✅ **SHIPPED**
The trust dial stops being a knob you set and becomes a level the DE earns. Research basis: **calibrated trust** (autonomy should track demonstrated capability, not vibes), **per-capability thresholds** (trust one action, not the whole agent), **eval-gated staging** (the Proving Ground is the promotion evidence), **promote slow / demote fast** (human approval on the way up, automatic and immediate on the way down). Migration 025: `trust_policies` (earned level 0–3 per action category, criteria jsonb, SELECT-only RLS), server-computed evidence from eval_runs + human_tasks outcomes + guardrail blocks (`compute_trust_evidence`), promotion via `request_trust_promotion` → `trust_promotion` human task → decideHumanTask hook #4 → `apply_trust_promotion` (self-approval blocked, evidence re-verified at apply time — stale regressions rejected), automatic one-level demotion on failing eval runs or guardrail blocks (audit `trust_demoted` + notice task, never below baseline). Manual raises above the earned level are audited as `trust_manual_override` and labeled in the UI so the earned path stays the celebrated one. Guardrails still cap everything — promotion widens autonomy only within them (016 composition rule untouched). *Honest limits:* evidence volume is thin until R1 activation (evals + human reviews + the guardrail record are the only sources today); guardrail demotion is tenant-wide v1.
**Moves scores:** Engine 8.5→9 (trust is earned machinery, not config), Features 7.5→8.

## R6 — Tenant playbook builder  ✅ **SHIPPED**

Definitions (`playbook_definitions`, migration 019) are validated server-side on publish (structured errors: unknown primitive, bad params, >20 steps, gate-ordering rules) and snapshotted into immutable `playbook_versions` — runs execute the snapshot via the generalized `playbook-execute` executor, never the live draft. Guardrail + trust-dial composition applies to every generated invoice exactly as in renewal_v1.
**Honest notes:** manual trigger only in v1 — `schedule`/`event` trigger types are reserved columns, not wired. Human-gate resume is split: `resume_playbook_on_task` advances post-gate steps it can do natively in SQL (guardrail_check / update_record / log_activity / complete); a post-gate `connector_action` needs HTTP, so the RPC parks the run in `resume_pending` and `decideHumanTask` fires the edge function's `advance` to finish it. `connector_action` degrades honestly — no connected connector / no target ref → step recorded `skipped`, run continues. `update_record` is a whitelisted status flip only. Legacy `renewal_v1` runs untouched (regression-verified).
**Moves scores:** Engine 8.5→9, Features 7.5→8.

## R7 — Scheduled & event playbook triggers  ✅ **SHIPPED**
Fills the `trigger_type` slots reserved in R6. Migration 020: `playbook_schedules` (friendly cadences — daily/weekly/monthly at a UTC hour, **deliberately not raw cron strings**; `next_fire_at` computed server-side on every write), `playbook_event_rules` (two event keys in v1: `invoice_overdue` w/ `overdue_days`, `ticket_synced_high_priority` w/ `priority`; per-target `cooldown_hours` dedup, default 24h), `playbook_trigger_fires` (firing log — every dispatch decision is a row: `pending_start` → `started`/`error`, or `skipped_dedup` recorded once per cooldown window). `dispatch_due_triggers()` (SECURITY DEFINER, pure SQL) evaluates due schedules + event matches into `pending_start` fires and stamps schedule `last/next_fire_at` atomically — a crashed HTTP processor can never double-fire a schedule. The `playbook-execute` `dispatch` action turns pending fires into real definition runs (same start path as manual runs — guardrails, gates, and the audit chain all apply) and audits each fire (`playbook_step`, `detail.kind='trigger_fire'`).
**Dispatch mode achieved (verified on the project):** pg_cron 1.6.4 + pg_net 0.20.3 + Vault all live → **primary: pg_cron job every 5 minutes** (`playbook-dispatch-5min` → `invoke_playbook_dispatch()` → pg_net POST with the Vault-held `playbook_dispatch_secret`, provisioned out of band, never in git). Live cron ticks observed in `cron.job_run_details` (succeeded). **Backup: opportunistic dispatch** when the Playbooks page loads, scoped server-side to the caller's tenant.
**Honest notes:** friendly cadences only, not cron expressions; two event keys in v1; events are *polled* on the 5-minute tick, not pushed; `all_eligible` targeting = accounts with a renewal date within N days (default 60). E2E verified: due schedule → fire `started` → run completed; overdue sent invoice → event fire → run; second dispatch inside cooldown → `skipped_dedup`; audit chain intact after all fires.
**Moves scores:** Engine 9→9.5 — playbooks now run without a human pressing the button, inside the same judgment machinery.

## Playbooks — "document that executes"  ✅ **SHIPPED**
Migration 031 + `playbook-execute` v8 + full `LivePlaybookBuilder.tsx` overhaul. Research basis: a linear step-list editor, NOT a node canvas (canvases become maintenance burdens for non-technical operators — the founder's own instruction, backed by SOP-tool adoption research); media embedded directly inside steps (SOP-tool adoption lesson: media as an afterthought/attachment kills adoption); instructions versioned and validated exactly like the existing typed-primitive playbook steps (same publish → immutable-snapshot doctrine as R6); branching rendered as simple indented lists, never canvas edges. Five new step types: **instruction** (markdown + embedded image/video, feeds an `instructions_context` accumulator for later ai/consult steps — dormant until LLM activation), **decision** (plain-language operator against an EARLIER step's output, one level of branch nesting, indented then/else rendering), **checklist** (human-tick list, rides the existing human-gate machinery as a new gate type), **wait** (parks the run, resumed by the EXISTING 5-minute dispatch cron — no new infrastructure), **sub_playbook** (runs another published playbook as a real child run, `parent_run_id`-linked, inheriting the parent's DE subject for data-access-grant enforcement — a child can never reach further than its parent). `de_playbook_charter`: the DE operating charter — which playbooks a DE runs and in what priority order; the dispatcher orders due schedules/event fires by charter priority (lowest first) when several playbooks match the same trigger. Builder UI: grouped step picker (Do something / Guide & explain / Flow control), inline plain-language validation errors, a dedicated "document that executes" SOP-style read view (numbered steps, rich instruction blocks with inline media, indented decision branches, action chips), and a server-side dry-run preview mode that simulates every write/connector/gate while running flow-control logic for real — no persistence, no external calls. Verified live end-to-end on a real tenant: all five new step types exercised together in one playbook (including both decision branches on separate runs), checklist human-gate resume, a REAL 65+ second timed wait resumed by the dispatch cron, and a sub_playbook child run whose inherited DE subject was correctly denied a connector call by data-access-grants (default-deny holds across the parent/child boundary). Negative tests all rejected as designed: 2-level decision nesting, forward-referencing decisions, unpublished/cyclical sub_playbook references. Priority-ordered dispatch proven by flipping two charter priorities and observing the evaluation order flip. Legacy `renewal_v1` regression-verified unaffected (auto-approved and gated-then-approved paths both still work). `verify_audit_chain` intact on both tenants touched. *Honest limits:* `instructions_context` is dormant until a consult step's LLM is activated; wait resume lands within the existing 0–5 minute dispatch-tick window, not to-the-second; decision nesting is a deliberate v1 simplicity cap, not a technical ceiling — deeper branching should be modeled as a sub_playbook.
