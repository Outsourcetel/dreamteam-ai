# 14 — Mission Delegation ("tell an employee to do a job in one sentence")

Status: **DESIGN OF RECORD — founder-approved 2026-07-22.** Build in flight.
Related: docs/11 (operating model), docs/12 (playbooks), north-star Phase B.

## The problem this solves

Every Digital Employee runs a standing profession (watchers → cases → SOP →
gates). What the platform could not do is take a one-sentence order —
*"Remy, run renewals for all Q3 accounts"*, *"Aria, implement the books for
client X"* — and turn it into governed, visible, stoppable work. Objectives
already exist as an arrival channel into the 5-minute work tick, but a
mission-sized directive entered the loop blind: no scope preview, no cost
estimate, no progress view, no kill switch. Missions add the steering wheel
and brakes to the engine that already runs.

## Founder decisions (locked 2026-07-22)

1. **Plan gate: ALWAYS.** No mission fans out until the founder has seen and
   approved the compiled plan (scope, procedure, gates, cost estimate).
   A future trust-dial notch may exempt tiny missions; not now.
2. **Approvals: one-by-one.** Gated items generated by a mission are reviewed
   individually at the Approvals desk. Bulk-approve is a future trust
   privilege, not a launch feature.
3. **Budget: soft warning.** A mission carries a cost estimate and warns when
   actual spend passes it; execution does NOT hard-stop. The tenant-wide AI
   budget (migration 100) remains the only hard ceiling.
4. **Rails built now**, while the platform's Anthropic org is offline. The
   compile step is the only brain-dependent stage; it returns an honest
   "brain offline" error until a key exists, and everything else is testable.

## Lifecycle

```
say it → compile (DE reads the order back) → founder approves the PLAN
       → fan-out under the standing rails → console (watch/pause/cancel)
       → completion report → optional graduation to standing watcher
```

1. **Say it** — a directive box on the DE's Employee File (and later the
   Workforce chat). Plain language, one sentence or several.
2. **Compile** — the DE's brain + its operating model produce a **Mission
   Plan**: interpretation; shape (below); scope as a *deterministic query*
   with resolved count and sample rows; per-case procedure (which
   SOP/playbook steps); which steps will stop at a human gate; cost estimate
   (LLM + outcome metering); dedup note (entities that already have open
   cases are listed and default-skipped). Compile writes NOTHING outside the
   mission row.
3. **Plan approval** — the founder edits scope (untick entities, change the
   window) and approves. Nothing else moves first. Status: `draft →
   awaiting_approval → approved`.
4. **Fan-out** — deterministic, no brain needed: mission → `de_objectives`
   (mission_id stamped) → cases (existing case facet, migs 225-228) →
   `de_work_items` → the standing 5-minute tick works them. **Missions
   inherit every rail: guardrails, trust dial, human gates, audit,
   metering. A mission can never bypass a gate.**
5. **Console** — on the Employee File: progress (done / in motion / waiting
   on you / blocked), spend vs estimate (warn-only), pause (stops claiming
   new work), cancel (cancels queued objectives/work items; never yanks an
   item mid-approval), and per-case links into the case timeline.
6. **Report** — when the last case settles: what was done, outcomes metered,
   exceptions, skips. If the mission was recurring in nature, the report
   offers **"make this standing"** → installs a watcher/schedule via the
   existing kit machinery. One-off order and permanent profession are the
   same thing at different settings.

## Mission shapes

- **batch** — fan out over N entities matching a filter (Remy: accounts
  renewing in Q3). Parallel cases, per-case gates.
- **project** — one subject, phased checklist, sequential with milestone
  gates (Aria: implement books for client X).
- **standing** — the directive is really a cadence ("post weekly") →
  compiles directly into a watcher/schedule PROPOSAL, not cases.

## Genericity rules (non-negotiable)

- ONE engine, zero role-specific code. Domain comes from the DE's operating
  model: its watchers, playbooks, scope vocabulary (which entity tables it
  works), desks, guardrails.
- The scope query is restricted to the DE's registered work sources —
  a marketing DE cannot scope over invoices.
- Compile uses the same model-routing and budget accounting as every other
  DE brain call.

## The operating-model object (prerequisite, audit gap #1)

`get_de_operating_model(de_id)` — a COMPOSED READ (no new state, SoR
principle): identity + archetype, work sources (watchers with label,
description, kind, next_fire_at, last_match_count), playbooks (published
definitions bound to the DE), desks/systems, cadence summary, guardrail
count, trust level. Two consumers: the mission compiler (server) and the
Employee File "How I operate" panel (founder legibility).

## Data (migration 248)

- `de_missions`: id, tenant_id, de_id, directive_text, status
  (`draft | compiling | awaiting_approval | approved | running | paused |
  done | cancelled | failed`), shape, compiled_plan jsonb (interpretation,
  scope_query, scope_preview, procedure, gates, estimate_usd, dedup),
  scope_edits jsonb, est_cost_usd, spent_usd, warn_sent boolean,
  created_by, approved_by, approved_at, finished_at, report jsonb,
  timestamps. RLS: tenant members read; owner/admin/manager write.
- `de_objectives.mission_id` (nullable FK) — fan-out attribution.
- `human_tasks.de_id` (nullable) — approval attribution (audit gap Q5:
  "Remy waits on you for 3 things"). Write sites updated: de-work
  escalations, browser-operator proposals, support draft gates.

## Edge function `de-mission`

Actions: `compile` (brain; honest error while org offline), `approve`
(server-side fan-out from the approved plan — deterministic), `pause`,
`resume`, `cancel`, `report`. All actions audit-logged. Fan-out is
idempotent per (mission_id, entity_ref) so a retried approve never
double-creates cases.

## Rollout

Land first on complete-loop DEs (Renewal, CS, SDR per the 2026-07-22
operating-model audit); other roles gain missions as their loops finish in
the locked execution order (Support → Renewal → Billing → Accounting →
Marketing). The directive box is visible for every DE from day one; compile
tells the truth about what a given DE can scope over.
