# Scalability Gaps Log — Multi-Tenant DE Platform

Running log of gaps surfaced while proving the 10-industry demo tenants (2026-07-18).
Each is a product/infra weakness that will bite at scale (many tenants, self-serve
onboarding). Kept so builds and fixes are *planned*, not reactive. Severity: P0 (blocks
scale / correctness) · P1 (degrades quality or ops) · P2 (polish / nice-to-have).

| # | Gap | Where it bit | Severity | Root cause | Proposed fix |
|---|-----|--------------|----------|-----------|--------------|
| G1 | **Bare-keyword guardrail patterns over-block** | masterclass DE refused its own refund-policy question; same bare `refund` rule in 9 tenants | P1 | A `blocked_phrase` guardrail with a single-word pattern (`refund`) matches ANY mention, so the DE can't even *explain* the policy it's meant to protect | Guardrail authoring: prefer phrase patterns; add a builder linter that warns on short/bare patterns; consider separating "must not SAY/PROMISE X" from "topic X" scope. Fixed live for the 9 tenants. |
| G2 | **URL ingestion fails on JS-rendered / WAF-protected help centers** | 11/41 sources returned HTTP 403 | P0 for onboarding | **PARTIALLY FIXED + diagnosed.** Empirically: header-hardening did NOT rescue the demo sources — a local curl gets 200 on masterclass.com/terms but the Supabase **edge fetch still 403s**, so these are **datacenter-IP / hard-WAF blocks**, not UA/header checks. A plain edge fetch fundamentally cannot pass a Cloudflare JS-challenge or an IP block. | **Shipped** `_shared/browserFetch.ts`: full browser header set + retry/backoff on 403/429/503 (rescues the real class of sites that block on UA/headers or briefly rate-limit), honest failure **classification** ("blocked — upload a file/PDF or connect via API"), and a **configurable render/proxy fallback** (`RENDER_FETCH_URL`/`RENDER_FETCH_HEADER`, dormant until an operator provisions a ScrapingBee/Browserless-style service — the only thing that defeats JS-challenge/IP blocks). Wired into `extract-document` (production) + `demo-ingest`. **Residual:** provision a render service, or use API connectors / file upload for hard-blocked sites. |
| G3 | **No PDF handling in the automated (dispatch) ingestion path** | mynd "Rent Ready Standards" PDF skipped | P1 | `demo-ingest` handles HTML only; `extract-document` handles PDF but is `verify_jwt`-only (needs a human browser session) | Give `extract-document` a service-role/dispatch mode, or fold unpdf into the dispatch ingestion path, so automated/connector ingestion can handle PDFs. |
| G4 | **Embedding is partial on larger docs — retrieval silently degrades** | first-community-cu: only 20/37 chunks embedded → its Fraud Prevention doc didn't retrieve → DE refused a legit fraud question and FAILED certification | P0 | The free in-runtime `gte-small` embedder caps how many chunks it embeds per `ingest-chunks` call (time budget); the rest are stored with null embeddings and rely on keyword-only match. No backfill, no visibility. | Add an embedding-backfill worker (cron) that finds `embedding IS NULL` chunks and embeds them in batches; surface an "embedded coverage %" per doc in Knowledge Quality; consider a paid embedding provider for large corpora. |
| G5 | **Golden references must be exhaustive or the judge false-negatives correct answers** | 3 tenants first "failed" certification because the DE gave MORE correct facts than the thin golden `expected_fragments`; judge scored grounded ~55 as "invention" | P1 | `eval-judge` grounds the answer against the supplied reference, treating extra (correct, in-KB) facts as unsupported | Judge should ground against the retrieved KB context, not only the golden reference; OR golden authoring guidance + a helper that drafts references from the KB. Meanwhile: write fuller references. |
| G6 | **Auth-login isolation not re-proven at the gotrue layer** | demo isolation proven at RLS (real uid via jwt claims) + DE layer, but not via a real password-grant login round-trip | P2 | Safety boundary — avoided re-entering demo passwords; vault demo password already deleted | For a full battery, script a password-grant login per admin in an isolated test harness (not in the assistant transcript), assert token→tenant scoping. |
| G7 | **PII/anti-phishing guardrails written as bare secret-noun patterns block the DE from GIVING security advice** | first-community-cu DE refused a legit "what do I do if I suspect fraud?" — its correct answer ("we'll never ask for *your password* or *one-time passcode*") matched the rule meant to stop the DE from *requesting* those secrets. Cost it a failed certification. | P0 (correctness) | The `blocked_phrase` pattern `your password\|your PIN\|one-time passcode` matches both a REQUEST for the secret and a mention/warning about it. Same root as G1 but higher stakes — it silences exactly the security guidance a support DE should give. | Guardrail intent needs direction: pattern should match a request verb ("what is your", "enter your", "tell me your", "provide your", "give me your" + secret), not the secret noun alone. Broadly: introduce an "intent-scoped" guardrail type (block when the DE ASKS for X) distinct from a topic block. Fixed live for first-community-cu. |

| G8 | **No headless/service run path for playbooks (testing + automation friction)** | Couldn't live-run the 50 demo playbooks without a service-role or user JWT; the operator service-key fetch is (correctly) classifier-gated, and no dispatch-secret path exists for `start`/`publish` | P1 | `playbook-execute` accepts `x-dispatch-secret` only for `action:'dispatch'` (cron); `start`/`publish`/`validate` require a full JWT. Great for security, but leaves no clean server-to-server way to run/validate a playbook in tests or automated onboarding | Add a narrow dispatch-secret path for `validate` + a service-triggered `start` (tenant_id explicit, same as de-answer/de-simulate already do) so QA harnesses and automated seeding can exercise playbooks without a browser login. |

## Notes
- All fixes above are tenant-agnostic (they help every future tenant), consistent with the genericity mandate.
- G2 + G4 are the two that most threaten *self-serve onboarding at scale* — a new customer's knowledge simply won't land reliably without them.
