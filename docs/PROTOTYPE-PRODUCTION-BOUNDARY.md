## 0. Production track status — Customer section (P1 + P2 SHIPPED)

**Platform-owner security model — "no one else has access unless I authorize it" (SHIPPED, migration 052).** The founder's own words, verbatim: "no one else, other than authorized by me as team member, ever has access to this platform... a panel to see my tenants and provision them... an easy way to remote access those tenants." Three pieces:

1. **`profiles.is_active` enforcement (real security gap, found and closed).** Before this fix, `is_active` was set by the UI's deactivate toggle but checked NOWHERE — no RLS, no RPC, no AuthContext check. "Deactivating" a team member was cosmetic; they could still log in and act normally. Fixed via a new `my_account_status()` RPC checked at session-restore, direct sign-in, and a new 60-second resync poll (so an already-open session is force-signed-out mid-session, not just blocked at next login).
2. **`platform_invites`** — the only owner-controlled path to platform-layer access (direct DB writes remain the only OTHER way, deliberately no self-serve platform signup). `invite_platform_team_member`/`list_platform_invites`/`revoke_platform_invite` (`is_platform_admin()`-gated) + `redeem_platform_invite` (any authenticated user with an existing profile, single-use, no parallel signup path). Deny-all RLS on the table itself — the invite code is a bearer credential, reads go through guarded RPCs only.
3. **Remote Access ("god-mode") — found disconnected, actually wired.** The existing `godModeSession`/banner/modal UI in `PlatformConsolePage.tsx` looked complete but its confirmation buttons never called `setGodModeSession` — clicking "Enter Tenant Workspace" did nothing. Fixed with real DB-gated `start_platform_remote_access`/`end_platform_remote_access` RPCs (durably audited in a new `platform_access_events` table) wired to real `enterRemoteAccess`/`exitRemoteAccess` AuthContext functions, called from every Remote Access button. UI copy renamed from "Support Access"/"God Mode" to "Remote Access" throughout (internal variable names unchanged).

Full detail, live verification evidence (invite create/reject/redeem/duplicate-reject/revoke, remote-access start/end with real audit rows shown, is_active flip-and-recheck, grant-check results): see the ROADMAP.md entry of the same name.

**FOUNDATIONAL: multi-level tenant hierarchy — "fix this once and for all" (SHIPPED, migration 050).** The founder's explicit, highest-priority architectural request, ahead of any new feature: a parent platform, tenants underneath it, and tenants able to have their own sub-tenants ("four levels is a possibility"), with a HYBRID creation model — self-serve where a tenant is trusted to do so, otherwise a creation request routes to the PLATFORM (not the parent tenant) for approval by default.

**Before this build:** the tenant model was strictly flat and two-level (platform → tenant, full stop). There was no `parent_tenant_id`, no concept of a sub-tenant, no way for a tenant like TCP to grant its own customers limited, customer-centric access as a nested workspace.

**After this build:** arbitrary-depth hierarchy. A sub-tenant is simply a `tenants` row with a `parent_tenant_id` pointer to another `tenants` row — recursively, to any depth.

**Why this was additive, not a retrofit (the load-bearing architectural insight):** every one of the ~87 existing `SECURITY DEFINER` functions and every existing RLS policy in this codebase isolates tenants strictly by `tenant_id`, with zero knowledge of hierarchy. Because a sub-tenant is *just* an ordinary `tenants` row, every one of those 87 functions continues to isolate a sub-tenant's own data exactly as strictly as it isolates any flat tenant today — **zero changes were made to any existing tenant-scoped RLS policy or function.** This was proven live, not just asserted: `resolve_access()` (the connector data-access-grant resolver, unmodified, migration 029) was called against a brand-new 3-levels-deep sub-tenant with no connectors and returned `connector_not_found`/`allowed:false` — byte-for-byte the same default-deny shape it returns for the flat top-level tenant Acme Telecom. No special-casing was needed, none was added.

**Schema (all additive):**
- `tenants.parent_tenant_id uuid references tenants(id)` (nullable; NULL = top-level tenant) + `tenants_not_self_parent` CHECK constraint (a tenant can never directly reference itself as its own parent — defense against the simplest cycle).
- `tenants.allow_self_serve_subtenants boolean default false` — governs whether creating a child under THIS tenant can skip platform approval. Default `false`, matching the earned-trust/"requires approval by default" pattern already used elsewhere in this codebase (trust dial, action execution).
- `tenant_ancestry` — a **maintained closure table** (`tenant_id, ancestor_id, depth`; `depth=0` is the self row), per the research grounding that hierarchical tenant isolation at scale should use a maintained closure table updated via trigger, not a recursive CTE evaluated per-row-check. A trigger (`trg_recompute_tenant_ancestry`, fires on insert/update of `parent_tenant_id`) walks the chain upward and rebuilds the affected tenant's row set, capped at depth 10 to catch any cycle that slipped past the CHECK constraint (e.g. via a race), and cascades the same recompute to every descendant when a mid-chain parent link changes. Every "is X an ancestor/descendant of Y" question is now an indexed equality lookup against this table, not a recomputed walk.
- Guarded helper RPCs — `is_ancestor_of`, `tenant_descendants`, `tenant_ancestors` — each rejects a caller with no legitimate relationship to the tenant in question (not a member, not a member of an ancestor, not a platform admin).
- `tenant_provisioning_requests` (id, requested_by_user_id, proposed_parent_tenant_id, proposed_name/industry, status pending/approved/rejected, reviewed_by, decided_at, rejection_reason, created_tenant_id) — reuses the established human_tasks/approval-request shape from `trust_promotion` (migration 025), adapted rather than forced: `human_tasks.tenant_id` is `NOT NULL` (confirmed against the live schema) and is strictly tenant-scoped, so platform-wide provisioning requests get their own lightweight table instead of bending a tenant-shaped table to fit a platform-shaped need.
- `feature_registry` (key, label, description, `default_enabled` — ships ON by default per the founder's explicit ask, category) + `tenant_feature_overrides` (tenant_id, feature_key, enabled, set_by, note) — the "control panel." `is_feature_enabled(tenant, key)` checks the override first, falls back to the registry default. Seeded with 7 representative entries (`connector_hub`, `proactive_triage`, `de_memory`, `finance_de`, `account_de`, `staleness_watchdog`, `identity_credential_inventory`) as a starting convention, not an exhaustive retrofit — new features should add a registry row going forward.

**RPC surface:**
- `request_subtenant(parent_tenant_id, name, industry)` — the single entry point. Caller must be `tenant_owner`/`tenant_admin` of the parent (or a platform admin creating a fresh top-level tenant with `parent_tenant_id=NULL`). If the parent has `allow_self_serve_subtenants=true` (or the caller is a platform admin): creates the tenant IMMEDIATELY, and additionally inserts a pre-approved (`status='approved'`) `tenant_provisioning_requests` row for audit symmetry — every tenant, self-serve or not, has exactly one traceable provisioning record. Otherwise: inserts a `status='pending'` row for the platform to review.
- `approve_subtenant_request(request_id)` / `reject_subtenant_request(request_id, reason)` — platform-admin-only (`is_platform_admin()`); approve creates the real tenant row and links the requesting user as owner if they have no tenant yet (mirroring `complete_signup`'s linking step, migration 049); reject records the reason and creates nothing.
- `guard_against_demo_tenant_parent` (trigger, mirrors migration 038's `guard_against_demo_tenant_assignment`) — the demo tenant (`a0000000-…0001`) can never be set as anyone's `parent_tenant_id`, and can never itself be given a `parent_tenant_id`. It stays a flat, isolated island exactly as it is today.

**A real bug found and fixed during this build's own live verification** (the "verify live, not just claimed" discipline this doc's other entries use): the first live test of `request_subtenant`'s self-serve path failed with `not a member of this tenant` from inside `append_audit_event` — that pre-existing function (unmodified, migration 002-era) requires the actor to be `service_role` OR a genuine member of the `tenant_id` being audited. The caller creating a brand-new sub-tenant is, by definition, never yet a member of that brand-new tenant, and a platform admin approving/rejecting a request is typically not a member of the parent tenant either. Fixed by auditing under the **parent** tenant's trail only when the caller is a genuine member of it, and skipping the tenant-scoped audit event entirely otherwise — the `tenant_provisioning_requests`/`tenant_feature_overrides` rows themselves (status, reviewed_by, decided_at, rejection_reason, set_by, note) are the durable record of the decision in that case, the same pattern already used for pure-platform actions elsewhere in this codebase (`platform_config_set`, migration 038, records nothing in `audit_events` either).

**Honest limits — explicitly deferred, named as follow-ups, not silently skipped:**
- **Rollup-visibility UI for a parent tenant viewing aggregate data across its own descendants** is a good candidate for a near-term follow-up — the read-only plumbing (`tenant_descendants`, opt-in ancestor-relationship checks in `caller_has_tenant_relationship`) is already in place, but no dashboard consumes it yet. The founder asked for the structure first; this is real, scoped-out future work, not a gap in the structure itself.
- **Tenant self-service feature-toggle UI** is deliberately out of scope — `set_tenant_feature_override` is platform-admin-only today, matching "each tenant has its own guardrails, security and permission levels" as something set from above, not self-serviced. A future tenant-facing subset of toggles (e.g. letting a tenant turn OFF a feature for itself, never ON one the platform hasn't enabled) is a reasonable extension.
- **The feature registry seed (7 entries) is representative, not exhaustive** by design — a lightweight "add a row when you ship a feature" convention going forward, not a retrofit of every capability shipped to date.
- `tsc --noEmit`: 0 errors. `vite build`: clean (pre-existing chunk-size warnings only, unrelated to this change).

**Research grounding (cited, not reinvented):** mature SLA/escalation systems (PagerDuty, Zendesk SLA policies, ServiceNow OLAs) use TIERED severity — a "breaching soon"/"at risk" warning BEFORE the hard breach, not a single alarm at one threshold. **Research grounding (cited, applied exactly, not reinvented):** current 2026 agent-memory research argues AGAINST summarizing at write time — "an agent that summarizes at write time collapses distinct episodes into semantic generalizations, destroying the episodic signal before it can be used." **Access-grant decision (deliberate, documented):** experience retrieval respects the SAME `data_access_grants` (029) ladder as everything else, via a new `resolve_category_access` RPC — the category-only sibling of `resolve_access` (029's own connector-specific lookup doesn't fit here, since experience is filed by category, not by a single connector; same ladder, same table, entered one level higher). **Hybrid Knowledge Retrieval — unifying two disconnected knowledge systems, found live during a founder product-demo walkthrough (SHIPPED):** this fix did not come from a prior audit pass — it was found by the founder and the assistant walking the actual customer-facing chat widget end-to-end, which is itself worth recording: live product walkthroughs surface a class of bug (two systems quietly diverging over time) that code-only audits tend to miss, because both systems individually looked fine in isolation.

**What was found:** the customer-facing widget path (`EndUserChatPage.tsx` → `runPortalTurn`/`draftAgentAction` in `src/lib/api.ts`) queried a LEGACY system — the `search_knowledge` RPC over `knowledge_articles`, pure Postgres full-text search (`ts_rank`/`websearch_to_tsquery`), no semantic understanding. Meanwhile the REAL production DE pipeline (`de-answer`, `widget-ask`, `specialist-consult`'s `resolve_inquiry`) had already been using a DIFFERENT, newer system — `knowledge_docs`/`knowledge_doc_chunks` with free 384-dim `gte-small` embeddings (Supabase.ai Session) via `match_doc_chunks` — genuine semantic search. These two systems had never been unified. Confirmed live: `knowledge_articles` held exactly 3 rows total, all for tenant `a1b2c3d4-…0001` (Acme Telecom), with its `embedding` column present in the schema but NEVER populated (0/3) — the table was already dead weight duplicating a system that worked better elsewhere. A second, independent gap was found in the same pass: `widget-ask`'s own header comment claimed to run "the same answer pipeline as de-answer," but it never actually computed an embedding or called `match_doc_chunks` at all — pure keyword `rankDocs()`, silently diverged from its own documentation.

**The founder's explicit ask, and the research behind the fix:** "a customer can ask a question in 10 different ways so it should be smart enough to understand that." Pure lexical search fails on vocabulary mismatch — "I'm locked out of my account" shares almost no words with a doc titled "SSO Login Troubleshooting" despite being the same issue. Pure semantic search alone can blur past a rare exact term a customer literally types (a product name, an error code). Same-day benchmark research cited here rather than reinvented: BM25-only and dense-embedding-only retrieval are statistically tied around NDCG ≈0.695–0.698, but a well-tuned HYBRID of both — combined via **Reciprocal Rank Fusion (RRF)**, which fuses RANK POSITION rather than raw score specifically to avoid the score-incompatibility problem between a bounded ts_rank value and an unbounded cosine distance — reaches NDCG ≈0.75, a ~7.4% lift over either alone. Current best practice is hybrid, not a choice between the two.

**The fix — one shared RPC, not two parallel systems:** migration 046 adds `hybrid_match_knowledge(p_tenant_id, p_query_text, p_account_id, p_query_embedding, p_match_count, p_subject_kind, p_subject_id, p_max_distance)`, extending `match_doc_chunks`'s existing scoping (tenant/account/subject/visibility/`is_current`, migrations 013/030/032) rather than duplicating it: a `search_tsv` generated column + GIN index was added to `knowledge_docs` (mirroring `knowledge_articles.search_tsv` from migration 038), a lexical CTE ranks docs by `ts_rank`, a semantic CTE ranks chunks by cosine distance, and the two rankings are fused via RRF (`score = 1/(60+lexical_rank) + 1/(60+semantic_rank)`, k=60 the standard constant from the source literature). `p_query_embedding` is nullable — a caller with no embedding available (the browser widget path; `gte-small` only runs inside the Supabase.ai edge runtime) still gets full lexical ranking; every edge function supplies a real embedding and gets the full hybrid fusion. `p_max_distance` (default 0.25) is a semantic relevance floor calibrated from a real measured gap during live testing (paraphrases of a covered question measured cosine distance 0.068–0.199; a genuinely unrelated question measured 0.29+) — without it, a single-doc tenant KB would return its nearest neighbor for ANY question, breaking honest escalation. **Consolidation, not addition:** `draftAgentAction` (the widget path) now calls the same `hybrid_match_knowledge` RPC instead of `search_knowledge`/`knowledge_articles` — only the retrieval step changed; the existing confidence-gating (`auditAnswer`, 0.55/0.75 thresholds) and escalation/persistence logic (`conversations`/`escalations` tables, `runPortalTurn`) were left untouched, per the explicit instruction not to touch machinery that already worked. `de-answer`, `widget-ask`, and both of `specialist-consult`'s retrieval spots were repointed at the same shared RPC; `widget-ask` gained real semantic retrieval for the first time (previously keyword-only despite its own comment claiming otherwise). A shared `_shared/knowledgeEmbed.ts` helper replaced three independent copy-pasted `embedText` implementations (`de-answer`, `specialist-consult`, and now `widget-ask`'s first one). `search_knowledge`/`knowledge_articles` were audited for any other frontend consumer before touching anything — grep across the whole repo found exactly one call site (this one) — so the legacy path was retired from the live retrieval flow rather than left running in parallel; the table and RPC were left in place undropped (harmless, already revoked from PUBLIC since migrations 038/040) rather than destructively dropped for 3 orphaned rows.

**Verified live (Acme Telecom, real "SSO Login Troubleshooting" doc, real `gte-small` embeddings via a temporary throwaway test function, deleted after verification):** four phrasings of the same underlying question — BEFORE (lexical-only, no embedding) matched only 1 of 4 (the title-adjacent phrasing, rank 1); the vague "I'm locked out of my account," the synonym "I can't sign in with my company credentials," and a typo'd "sso lgin brokenn after update" all returned **zero matches** (would have escalated). A genuinely unrelated question ("weather in Tokyo... ramen shop") correctly returned NO match after the relevance floor was added (it had incorrectly matched at distance 0.29 before the floor existed — caught and fixed during this same verification pass) — honest escalation behavior is preserved, not just claimed.

## 4. Gate criteria — when to start the production track

Start the production build when **all** of these are true:

1. **Product model is stable** — Phases 4–7 complete and the entity/outcome + DE profile design has survived at least 3 real prospect walkthroughs without structural change requests.
2. **A committed design partner exists** — at least one customer with a signed pilot (or strong LOI) willing to run real data through the system.
3. **The core loop is specified end-to-end on paper** — for one concrete use case (e.g. support ticket resolution): data in → DE acts → guardrails checked → human gate → audit written. No unresolved "TBD" in that chain.
4. **Budget/runway supports a 4–6 month backend build** — multi-tenant schema, API layer, LLM orchestration, audit infrastructure, security review.

## 5. Production build — what gets built fresh (not migrated)

- Multi-tenant Postgres schema with row-level security; **no cross-tenant access, ever**
- API layer between UI and data (UI never queries tables directly)
- LLM orchestration: provider-abstracted, confidence scoring, fallback routing
- Immutable audit log (append-only), guardrail versioning
- Real ingestion pipeline (embeddings, chunking, citations)
- Observability, rate limiting, SSO/SCIM, pen-test before first real tenant

The frontend is **rewired to real APIs, not rewritten**. Estimated reuse: most of the UI, ~all of the domain model, none of the data layer.
