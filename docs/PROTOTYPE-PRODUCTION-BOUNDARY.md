**Research grounding (cited, not reinvented):** mature SLA/escalation systems (PagerDuty, Zendesk SLA policies, ServiceNow OLAs) use TIERED severity ‚Äî a "breaching soon"/"at risk" warning BEFORE the hard breach, not a single alarm at one threshold. Escalation-loop prevention in those same systems relies on a cooldown so the same stale item doesn't re-fire on every evaluation tick forever. Both principles are implemented directly: two tiers (`warning`, `breach`), independently configurable per (tenant, target_kind), and a unique-constraint-backed cooldown so a stale item escalates once per tier and then goes quiet until the tier changes or the item is resolved.

Schema: migration 042 ‚Äî `staleness_policies` (tenant_id, `target_kind` ‚Äî a deliberately free-text configuration key, NOT a widened enum, the same "no enum widening for new domains" lesson migration 036 already applied to `source_category` ‚Äî `warning_after`/`breach_after` intervals, `enabled`; seeded per-tenant with sensible platform defaults: `onboarding_project` 14-day warning / 30-day breach, `pending_review_task` 24-hour warning / 72-hour breach; tenant-overridable via ordinary UPDATE through the existing tenant-scoped RLS policy, no new RPC needed). `staleness_escalations` ‚Äî the tiered, deduplicated ledger: one row per (tenant, target_kind, target_id, tier); a partial unique index on unresolved rows IS the cooldown mechanism (only one open escalation per exact tier per target at a time; a tier change is a different row, not blocked; a resolved-then-restale item can escalate again fresh). `check_staleness(tenant_id)` ‚Äî the primitive itself: SECURITY DEFINER, evaluates every enabled policy with an EXPLICIT `tenant_id` filter on every query (no reliance on RLS ‚Äî this runs via the cron dispatcher, outside RLS, exactly like `dispatch_due_triggers`/`poll_de_work_sources_targets`), finds stale targets past warning/breach thresholds with no open escalation yet at that tier, creates the escalation plus a plain-language `human_task` ("This onboarding project for Acme Telecom hasn't been touched in 16.0 days (warning threshold: 14.0 days)."), and resolves open escalations whose underlying item is no longer stale.

## 4. Gate criteria ‚Äî when to start the production track

Start the production build when **all** of these are true:
